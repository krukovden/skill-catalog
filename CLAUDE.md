# SkillCatalog — repository rules

A platform-agnostic catalog of skills for Claude Code, GitHub Copilot, and OpenAI Codex.
This file is the contract for extending it. Read it before adding, moving, or removing a
skill.

## The one-line model

`skills/<bucket>/<name>/SKILL.md` is the **only** hand-written source of truth. Everything
else — `catalog.json`, `.claude-plugin/marketplace.json`, `.claude-plugin/plugin.json` — is
generated by `npm run build`. Never edit a generated file by hand; change `skills/` and
rebuild.

## Buckets decide what ships

Every skill lives in a bucket folder. **The bucket is the status** — there is no separate
`status:` field to keep in sync, and there must never be one.

| Bucket | Promoted? | Meaning |
|--------|-----------|---------|
| `engineering/` | yes | day-to-day code work |
| `ops/` | yes | machines and infrastructure |
| `productivity/` | yes | workflow tools not tied to code |
| `in-progress/` | no | being built; committable but never shipped |
| `deprecated/` | no | retired; kept for the record |

Promoted skills appear in `catalog.json`, in `.claude-plugin/plugin.json`'s `skills` array,
and in the root `README.md`. Unpromoted skills appear in **none** of the three. Promoting,
demoting, or retiring a skill is a `git mv` between buckets plus `npm run build` — nothing
else.

The bucket list lives in `cli/catalog.js` (`PROMOTED_BUCKETS` / `UNPROMOTED_BUCKETS`) and
nowhere else. To add a topical bucket: add it to `PROMOTED_BUCKETS`, create the folder with
a `README.md`, rebuild.

## Invariants

Hold all of these, or the build and tests will disagree with you:

1. **Frontmatter `name` matches the folder name.** `loadSkillFromDir` throws otherwise.
2. **Every skill has `name` and `description` in frontmatter.** `description` is what the
   model reads to decide whether to fire the skill, so it carries the trigger phrasing
   ("Use when the user…"), not a human-facing blurb.
3. **A skill directory is self-contained.** No `../` references into another skill. Helper
   scripts go in `scripts/`, on-demand docs in `references/`, eval prompts in `evals/`.
   Adapters copy the directory verbatim; anything outside it will not arrive.
4. **Every bucket has a `README.md`** listing its skills, each name linked to its
   `SKILL.md`. It also keeps otherwise-empty buckets in git.
5. **Promoted skills appear in the root `README.md` skill table**; unpromoted ones do not.
6. **`package.json`'s `version` is the release version.** `build.js` reads it and writes it
   into `plugin.json` and `catalog.json`. Bump it on every user-visible change — Claude
   compares `plugin.json`'s `version` against the installed copy to decide whether users
   see an update, so a version that does not move means installed plugins never update.
7. **Nothing under `skills/` sits outside a bucket.** `scanSkills()` warns loudly if it
   does, because such a skill silently disappears from every artifact.

## After changing anything under `skills/`

```bash
npm run build     # regenerates catalog.json + .claude-plugin/*
npm test          # adapter unit tests + build sanity check
git status        # the generated files are committed — review their diff too
```

Both must be clean before committing. `npm run build` exits non-zero if no promoted skill
is found, and prints a `⚠` for every skill it skipped — treat those warnings as failures.

## Platform support lives in adapters, not in skills

`cli/adapters/{claude,copilot,codex}.js` own everything platform-specific. A skill author
writes one `SKILL.md` and never thinks about platforms; adding a fourth platform is one new
adapter file and zero changes to any skill.

Adapters are pure where it matters: `outputs(skill)` returns what *would* be written and is
what the tests assert against; `install()` performs the writes. Keep that split — a new
adapter that only implements `install()` is untestable.

Two behaviours are load-bearing and easy to break:

- **File modes are preserved on copy.** Skills ship executable helper scripts; installing
  them as `0644` makes the skill fail at runtime with "permission denied".
- **The Codex `AGENTS.md` block is idempotent.** It is delimited by
  `<!-- skillcatalog:start -->` / `<!-- skillcatalog:end -->` and only that block is
  rewritten; unrelated user content in `AGENTS.md` must survive re-installs.

## Evals

`skills/<bucket>/<name>/evals/` holds the prompts that check a skill actually helps.
Run artifacts go to `skills/<bucket>/<name>-workspace/`, which is git-ignored — results are
transient and must never be committed. When changing a skill's `description` (its trigger),
re-run its trigger eval: a description edit is a behaviour change, not a docs change.

## Adapted third-party skills

Some skills are adapted from other people's work. When you add or materially change one,
record it in `ATTRIBUTION.md` at the repo root — what it derives from, and how it differs —
and link that file from the bucket `README.md`. Keeping the licence notice with the derived
work is a condition of using it, not a courtesy.

## Local-only

`docs/` is git-ignored: design notes that are never published. Do not put anything there
that the build or another contributor needs.
